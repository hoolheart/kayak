#### 3.2.2 TC-S1-010-05 预期结果不一致

**位置**: 第254-258行

**问题**: 测试用例允许返回 401 或 404 两种状态码，应明确唯一正确答案。

**当前描述**:
```
| HTTP状态码 | 401 或 404 |
| code | 401 或 404 |
```

**分析**:
- Token有效但用户被删除时，JWT验证会通过（Token signature valid）
- 但查询数据库用户不存在
- 从安全角度，返回404会暴露用户存在过的事实
- 返回401更安全，但不符合RESTful规范

**建议**: 明确为一种状态码，建议 **401 Unauthorized**，原因：
1. 避免信息泄露（不暴露用户是否存在）
2. 与其他认证失败场景一致
3. TC-S1-010-21 也应统一

**优先级**: P2 (建议性)

---

#### 3.2.3 TC-S1-010-13 用户名唯一性状态码建议

**位置**: 第592-593行

**问题**: 测试用例允许返回 409 或 400，建议统一。

**当前描述**:
```
| HTTP状态码 | 409 Conflict 或 400 |
```

**建议**: 使用 **409 Conflict** 更符合语义，因为：
1. 资源冲突（用户名已被占用）是明确的业务逻辑错误
2. 400 Bad Request 通常用于请求格式或验证错误
3. RESTful API 规范中 409 表示与服务器状态冲突

**优先级**: P3 (低)

---

#### 3.2.4 安全测试场景缺少独立测试用例ID

**位置**: 第3.1节、第3.2节

**问题**: 授权测试场景（3.1 用户只能访问自己的数据、3.2 跨用户头像访问）没有分配具体测试用例ID。

**建议**: 为这些安全场景分配正式测试用例ID：
- TC-S1-010-25: 用户只能访问自己的数据
- TC-S1-010-26: 跨用户头像访问控制

**优先级**: P2 (建议性)

---

#### 3.2.5 TC-S1-010-22 修改密码后Token有效性测试价值存疑

**位置**: 第964-995行

**问题**: 该测试验证修改密码后现有Token仍然有效。

**分析**:
- 从安全角度，某些系统会在密码修改后使所有会话失效（Session Invalidation）
- 但该测试验收标准未要求此行为
- 测试本身设计合理

**建议**: 明确修改密码后Token的处理策略：
1. 如果修改密码后Token应继续有效：测试正确
2. 如果修改密码后应使所有其他Token失效：需要补充测试验证

**优先级**: P3 (低)
