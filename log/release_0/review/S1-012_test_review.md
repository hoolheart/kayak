## 发现的问题

### 1. TC-S1-012-03: 测试代码与测试名称不完全匹配 (轻微)

**问题描述**:  
测试名称为"刷新后恢复用户信息测试"，但自动化代码中并未真正模拟页面刷新/应用重启，只是设置了存储值并检查恢复。真正的集成测试是 TC-S1-012-02。

**建议**:  
- 选项A: 修改测试名称为"用户信息本地存储与恢复测试"
- 选项B: 在测试步骤中明确说明这是单元测试级别，不包含真正的应用重启

**严重程度**: 低

---

### 2. TC-S1-012-11: 优先级可能偏低

**问题描述**:  
"登录后返回原始请求页面"是一个重要的用户体验功能，但被标记为 P2。

**建议**:  
考虑将优先级提升至 P1，因为这是一个常见的用户交互场景。

**严重程度**: 低

---

### 3. TC-S1-012-14: 缺少对 Refresh Token 过期时具体错误类型的验证

**问题描述**:  
测试步骤提到"刷新Token失败（401）"，但测试代码只验证了 `UnauthizedException()`。当 Refresh Token 过期时，服务器可能返回不同的错误码或消息。

**建议**:  
增加对具体错误类型的验证，例如：
```dart
when(mockAuthApi.refreshToken('expired_refresh'))
    .thenThrow(UnauthizedException(code: 'REFRESH_TOKEN_EXPIRED'));
```

**严重程度**: 中

---

### 4. TC-S1-012-15: Token刷新间隔限制的时间假设

**问题描述**:  
测试使用 `Duration(milliseconds: 100)` 作为短时间判断，但未说明这个时间窗口的来源。如果刷新间隔是 30 秒，100ms 测试可能无法准确验证防抖逻辑。

**建议**:  
在测试用例的备注中说明刷新间隔的配置值，或使用配置值进行测试。

**严重程度**: 低

---

### 5. TC-S1-012-17: 连续刷新场景的测试断言不完整

**问题描述**:  
测试验证了按顺序调用两次 refreshToken，但没有验证最后一次调用后存储的 token 是否正确更新。

**建议**:  
增加断言验证 refresh 后的 token 存储状态：
```dart
await tokenManager.refresh();
expect(tokenManager.currentAccessToken, equals('access_2'));
expect(tokenManager.currentRefreshToken, equals('refresh_2'));
```

**严重程度**: 低

## 建议改进项

### 建议1: 增加异常场景测试 (建议添加)

当前测试覆盖了成功路径和基本的失败路径，但以下场景未被测试：

1. **Token存储写入失败**: 当 SecureStorage.write() 抛出异常时的处理
2. **Token解析失败**: 存储的 token 格式损坏或无法解析
3. **网络超时**: 刷新 token 时网络请求超时

**影响**: 低 - 这些是防御性编程场景，主流程测试已覆盖

---

### 建议2: 测试用例文档增强

2. **测试代码片段**: 建议所有自动化代码使用相同的变量命名约定（如 `testUser` 应在测试开头定义）
