## 2. 详细评审意见

### 2.1 必须修复 (Blockers)

#### 🔴 [BLOCK-001] R2-S1-001-C: HDF5 时序数据查询 API — 工时严重不足

**问题描述**: 预估工时 **6h** 明显不足。该任务包含：
1. 创建 `experiments/data.rs` handler 模块
2. 实现 HDF5 文件按时间范围切片读取
3. **自研 LTTB 降采样算法**（Rust 生态无成熟 crate）
4. 查询参数解析和验证
5. 路由注册和响应格式封装

PRD 第 2.1.2 节明确承认："Rust 生态中无可用的成熟 LTTB crate，需自行实现（预估额外 2-4h 工作量）"。即使取最小值 2h，剩余 4h 需要完成 HDF5 读取、API handler、DTO 定义、参数校验、错误处理、单元测试，这对于生产级代码过于紧张。

**影响**: LTTB 算法实现若被压缩，可能导致降采样精度不达标或边界情况处理缺失；HDF5 文件读取的错误处理（文件不存在、损坏、并发访问）可能不完备。

**修改建议**:
- 将 R2-S1-001-C 工时从 **6h 调整为 8~10h**
- 或将 LTTB 算法实现拆分为独立子任务 R2-S1-001-C2 (2h)，由 sw-tom 在详细设计阶段先行完成算法原型验证
- 在详细设计文档中强制要求包含 LTTB 算法的单元测试用例设计（覆盖空数据、单点数据、已降采样数据不二次降采样等边界）

**相关依据**: PRD v2.0 §2.1.2; 可行性评估报告 §3.4 (数据分析模块重估 +15%)

---

#### 🔴 [BLOCK-002] R2-S2-003-C: Python SDK 开发实现 — 工时严重不足

**问题描述**: 预估工时 **10h** 严重不足。该任务要求从零开始构建完整 Python SDK：
1. 清理旧代码（实际上 `kayak-python-client/` 目录只有 `.gitignore`，无旧代码）
2. `KayakClient` 主类 + 上下文管理器支持
3. `AuthManager`（Token 自动刷新，含过期前 5 分钟刷新逻辑）
4. 设备/测点/试验接口模块（至少 3 个资源模块）
5. 数据下载模块（HDF5 下载 + pandas/numpy 转换）
6. 异常体系（5 个异常类层次结构）
7. 示例脚本 + `pyproject.toml`

可行性评估报告 §3.5 将 Python SDK 重估为 **52h**（核心 26h + 数据下载 18h + 发布 8h）。当前任务 R2-S2-003-C 试图在 10h 内完成核心+数据下载的绝大部分工作，相当于压缩了约 70% 的工时。

**影响**: SDK 将成为 Sprint 2 的最大延期风险点。Token 自动刷新逻辑、HDF5 到 pandas 的时间戳索引处理、边界错误处理都可能被牺牲。

**修改建议**:
- 将 R2-S2-003-C 工时从 **10h 调整为 14~16h**
- 或进行范围削减：将数据下载/转换（`to_dataframe()` / `to_numpy()`）移至 R2-S2-003-F (后续迭代)，Sprint 2 仅完成核心认证 + 设备/测点/试验查询接口
- 明确 `pyproject.toml` 使用 Poetry 还是 setuptools（PRD 未明确，需在详细设计中确定）
- 删除 "清理现有旧代码" 描述（目录为空，无代码可清理）

**相关依据**: 可行性评估报告 §3.5, §7.1 RISK-04; PRD v2.0 §2.4.2

#### 🔴 [BLOCK-003] R2-S2-002-D: 团队管理前端 — 工时偏紧且范围过载

**问题描述**: 预估工时 **10h** 内包含：
1. `features/teams/` 模块搭建
2. 团队列表页面
3. 团队详情/设置页面
4. 成员管理页面
5. **AppBar 团队选择器组件**（全局组件，影响 AppShell）
6. 资源创建时的归属选择（需修改现有工作台/方法/试验创建对话框）
7. **团队切换后的页面重定向逻辑**（跨多个模块的复杂状态管理）
8. 更新侧边导航

其中 "团队切换后的页面重定向逻辑" 涉及全局状态（当前选中团队）和跨页面协调。当用户从团队 A 切换到团队 B 时，若当前页面正在查看团队 A 的专属资源（如 `/workbenches/a-uuid`），需要：
- 检测资源是否在新团队范围内
- 决定是否重定向到列表页
- 保持其他页面状态不丢失（PRD §3.2 可靠性需求）

这个逻辑需要深入修改现有的路由守卫或全局状态管理，10h 过于乐观。

**影响**: 团队切换重定向逻辑若实现不完整，用户可能看到"资源不存在"错误或无提示的空白页面。

**修改建议**:
- 将 R2-S2-002-D 工时从 **10h 调整为 12h**
- 或将 AppBar 团队选择器拆分为独立子任务（2h），因为它涉及 `AppShell` 重构
- 在详细设计 R2-S2-002-C 中强制要求：明确团队切换状态的持久化方案（Provider 作用域、与现有 Auth Provider 的交互、切换时是否刷新所有页面数据）

**相关依据**: PRD v2.0 §2.3.3; 可行性评估报告 §3.6 (团队管理 UI 重估 28h)

### 2.2 建议修改 (Recommendations)

#### 🟡 [REC-001] 明确 experiments 表的 Migration 方案

**问题描述**: R2-S2-001-C 提到 "为 experiments 表添加 owner_type + owner_id 字段"，PRD §6.1 给出了 ALTER TABLE SQL。但项目使用 sqlx 的 migrate 功能（见 `Cargo.toml` 中 `features = ["migrate"]`），目前无独立的 `migrations/` 目录（使用嵌入式迁移或运行时迁移）。

任务文档未明确：
- Migration 文件存放位置（`migrations/` 目录需要新建）
- 迁移执行方式（应用启动时自动执行 vs 手动执行）
- `owner_id` 的 DEFAULT 值处理（PRD 建议 `DEFAULT user_id`，但 SQLite ALTER TABLE 对 DEFAULT 的限制较多）

**修改建议**:
- 在 R2-S2-001-B（详细设计）中增加 Migration 策略设计条目：
  - 使用 `sqlx migrate add` 创建迁移文件
  - 迁移脚本分两步：(1) 添加列允许 NULL → (2) UPDATE 填充现有数据 → (3) 添加 NOT NULL 约束
  - 应用启动时自动执行 `sqlx::migrate!().run(&pool)`
- 在 R2-S2-001-C 验收标准中增加："Migration 可回滚（`sqlx migrate revert` 测试通过）"

#### 🟡 [REC-002] 增加 HDF5 并发读取安全机制任务

**问题描述**: PRD §3.2 可靠性需求明确要求："HDF5 并发读取无冲突"，策略包括"试验结束后标记 HDF5 文件为'可安全读取'"和"文件快照副本策略"。但 tasks.md 中没有对应的后端开发任务来处理这个问题。

当用户通过分析页面查询正在进行中的试验数据时，后端可能同时正在写入 HDF5，会导致读取冲突或损坏。

**修改建议**:
- 在 R2-S1-001-B（详细设计）中增加 HDF5 并发读取安全设计
- 在 R2-S1-001-C（开发实现）中增加：
  - 试验状态检查（仅 `completed` / `error` 状态的试验允许查询，或 `running` 时使用只读打开 + 文件锁重试）
  - 或实现临时文件快照副本策略（复制到 `/tmp/` 后读取）
- 或者明确将此风险标记为已知问题，在 Sprint 1 中仅支持已完成试验的数据查询，后续 Sprint 再实现并发安全

### 2.3 疑问 (Questions)

#### ❓ [Q-001] LTTB 降采样算法的精度验证标准是什么？

**问题描述**: R2-S1-001-A 测试用例提到 "降采样功能测试（验证点数为 downsample 参数值）"，但 LTTB 算法在数据点少于 downsample 时通常返回全部数据。验收标准 "降采样后点数符合 downsample 参数" 表述模糊。

**需要澄清**:
- 当原始数据点数 < downsample 时，是否返回全部原始数据？
- 当原始数据点数 == downsample 时，是否还需要降采样？
- 降采样后点数的精确要求：`<= downsample` 还是 `== downsample`（当数据充足时）？
- 是否需要验证降采样后的数据分布保留原始数据特征（如极值点保留）？

**建议**: 在 R2-S1-001-A 测试用例设计中明确边界条件，并在 R2-S1-001-B 详细设计中给出 LTTB 伪代码和点数计算规则。

---

#### ❓ [Q-002] 团队资源查询 API 的现有查询参数扩展方案

**问题描述**: PRD §2.2.3 提到 `GET /api/v1/workbenches?scope=team&team_id={id}`，但 tasks.md 中没有明确的任务来修改现有的 `workbenches` GET handler。

R2-S2-001-C 提到 "实现 Team Service（CRUD + 邀请逻辑）" 和 "实现 Team API handlers（所有 REST 端点）"，但现有的工作台/方法/试验列表查询 handler 的修改（增加 `scope` / `team_id` 查询参数过滤）归属不清晰。

**需要澄清**:
- 现有资源列表 API 的修改是否包含在 R2-S2-001-C 中？
- 还是作为 R2-S2-001-D 代码审查的发现项补充？
- 前端在团队切换后调用现有 `/api/v1/workbenches` 还是新增 `/api/v1/teams/{id}/workbenches`？

**建议**: 在 R2-S2-001-B 详细设计中明确团队资源查询的 API 方案（查询参数扩展 vs 嵌套路由），并相应调整 R2-S2-001-C 的任务描述和工时。

## 7. 特别风险提示

### 7.1 fl_chart Web 性能风险

PRD R2-RISK-01 已标记此风险。`fl_chart` 在 Flutter Web 上处理 10,000+ 数据点的交互性能未经项目验证。虽然降采样可将点数控制在 1,000 以内，但多曲线（≥4 条）叠加时的渲染性能仍有不确定性。

**缓解**: R2-S1-002-D 开发中应预留 1~2h 用于性能基准测试，若 fl_chart 不满足要求，可快速降级为 CustomPainter 实现（但此降级方案需要额外 4~6h，当前工时未预留）。

### 7.2 团队管理数据迁移风险

PRD R2-RISK-04 已标记此风险。`experiments` 表添加 `owner_type` + `owner_id` 字段时，若 Migration 设计不当（如直接添加 NOT NULL 列而无 DEFAULT），会导致现有用户数据表损坏。

**缓解**: 必须在 R2-S2-001-B 中设计分步迁移脚本，并在 R2-S2-001-E 中执行 Migration 回滚测试。
